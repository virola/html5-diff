摘要

HTML是万维网的核心语言。 W3C发布HTML5标准和HTML5.1标准。 WHATWG由发布了另外一个标准HTML，这是HTML5.1的超集。 本文将会介绍HTML5标准和HTML4标准的差异，并列举出一些案例。 由于标准本身仍然还在开发中，所以本文可能不会提供标准的精确信息。 如果有所疑惑，随时查看标准原文[HTML5] [HTML5NIGHTLY] [HTML]

1 引言

1.1 这篇文档的使用范围

这篇文档包含了W3C HTML5规范， W3C HTML5.1规范和WHATWG HTML标准。 为了便于阅读，它们被归类为“看似统一”的规范：“HTML规范”或者就叫“HTML”。 当某些概念在这三种标准中相一致时，我们就这样简要地称呼它们；否则，我们会分别指出它们的不同。

1.2 HTML的历史

HTML自从上世纪90年代早期被引入因特网，一直在不断地演进。 HTML的部分功能是通过标准引入的；其他则是在软件实现中引入的。 在某些方面，实现和Web开发实践相互融合对方的规格和标准；但在其他方面，他们又存在分歧。

HTML4在1997年成为W3C的推荐标准。 它继续充当了HTML核心功能的一份概要指南，但它没有为构建相互操作的网提供充足的信息，尤其是在网页内容。 XHTML1和DOM Level2 HTML都有相同的问题。 XHMTL1定义了HTML的XML序列化规范，而DOM Level2 HTML定义了HTML和XHTML的JavaScript API。 新HTML规范替代了这些文档 [DOM2HTML] [HTML4] [XHTML1]

HTML规范反映出了自2004年开始人们对研究现代浏览器和网页内容的努力。这份规范：

  1. 定义了一种叫作HTML的单一语言，它可以使用HTML语法和XML语法进行编写。

  2. 定义了详细的处理模型，以更快地实现交互操作

  3. 改进了文档的标签

  4. 为新兴的成语引入标签和API，例如Web应用

1.3 开放性问题

HTML的内容，以及依赖HTML的这篇文档，仍然在HTML工作组和WHATWG邮件列表上被讨论。

WHATWG HTML的开放性问题，请参见标准顶部的“Issues”部分。标准的边缘还有注解盒子指向这个BUG。

W3C HTML5和W3C HTML5.1的开放性问题，请参看“文档的状态”章节

1.4 向后兼容性

HTML是通过使用客户端的方式来完成向后兼容的。为了保持这门语言对于开发者相对简单，一些旧的元素和属性并没有被包含在内，例如这章文档其他章节中所列举的那些表象的元素最好还是通过CSS来处理。

客户端不管怎样还必须去支持那些旧的元素和样式。 这就是为什么HTML规范为Web开发者清楚地分离了必备条件和客户端样式表；举例来说，这意味着Web开发者不能使用isindex或者是 plaintext 元素，但是客户端却必须支持它们，以兼容原有Web内容中的这些元素。

既然HTML已经将对Web开发者的一致性要求与客户端分离，也就没有添加“deprecated”标识的必要了

1.5 开发模式

HTML4标准在被客户端完全实现之前就达到了推荐状态。HTML4仍然没有被完全实现，因为它包含了一大堆的错误。现在这些错误已经在HTML5中修复了，客户端的实现也比HTML4时更贴近标准。

WHATWG HTML标准仍处于持续的开发之中，因此随着时间的推进，错误被修复，新的功能也被引入。一些功能也可以被从标准中移除，假如开发者们缺少实现的兴趣，或者Web开发者们不使用，或者其他原因。WHATWG不会发布冻结快照。

W3C HTML5标准正在努力达到推荐状态。 大体上不会引入新的功能，除非它们已经被至少两个浏览器实现了并且通过了交互演示测试。小错误的修正可能被应用。这意味着W3C HTML可能会包含WHATWG和（或）W3C HTML5.1中的错误。

W3C HTML5.1标准与WHATWG HTML比较相似：W3C HTML5.1标准仍处于持续的开发之中，因此随着时间的推进，错误被修复，新的功能也被引入，功能也可能会被移除，因为WHATWG同样的原因。它会择优来自WHATWG的改变，也会获取直接改变。我们期待它最终能够通过和W3C HTML5一样的开发过程，然后我们可以得到一个新的版本。

2 语法

HTML定义了一种语法，被称为HTML语法。虽然与网络上发布的HTML4和XHTML1的绝大部分内容是兼容的，但HTML语法并不兼容HTML4中更难懂的SMGL功能。例如 processing instructions 和 shorthand markup 因为大多数的浏览器都不支持这些功能。使用HTML语法编写的文档需要以text/html媒体类型来提供。

HTML还为HTML语法定义了详细的解析规则（包括错误处理），绝大部分与HTML4-era实现是兼容的。客户端需要对那些媒体类型为text/html的资源应用这些规则。这是一份依照HTML语法规则编写的文档示例：

<!doctype html>
<html>
  <head>
    <meta charset="UTF-8">
    <title>Example document</title>
  </head>
  <body>
    <p>Example paragraph</p>
  </body>
</html>

其他可用于HTML的是XML。 这种语法与XHTML1文档和实现相兼容。 使用这种语法的文档需要以XML媒体类型（例如application/xhtml+xml或者是application/xml）来提供， 而且元素需要被放置在命名空间 http://www.w3.org/1999/xhtml中并且遵循XML规范所提出的规则。[XML] [XMLNS]

下边是一个依照XHTML语法编写的一个HTML示例文档：

<?xml version="1.0" encoding="UTF-8"?>
<html xmlns="http://www.w3.org/1999/xhtml">
  <head>
    <title>Example document</title>
  </head>
  <body>
    <p>Example paragraph</p>
  </body>
</html>

2.1 字符编码

对于HTML语法， 网页开者人员必须声明字符编码。 有三种方式可以完成：

  • 对于HTML语法， 网页开者人员必须声明字符编码。 有三种方式可以完成：Content-Type

  • 在文件的开始使用一个Unicode字符序标记(BOM)字符。 这个字符提供了一份供编码使用的签名。

  • 使用一个带有charset属性的meta元素。在文档的前1024个字节中指定这个属性来指定文档的编码。例如，<meta charset="UTF-8">可以来用指定文档的字符编码为UTF-8。这方法明显更方便，取代了原来的 <meta http-equiv="Content-Type" content="text/html; charset=UTF-8"> 但这种语法仍然是允许的。

在XML语法中，任何一种DOCTYPE声明都是可以使用的，或者也可以被省略。带有XML媒体类型的文档一直都会被以标准模式处理。

2.2 文档类型

HTML语法需要一个doctype来确保浏览器使用标准模型来渲染页面。 doctype没有其他的用途。[DOCTYPE]

DOCTYPE对HTML的定义是<!DOCTYPE html>并且是大小写不敏感的。 之前版本HTML的DOCTYPES更长一些，这是因为HTML语言是基于SGML的，因此需要有一个DTD引用。 现在已经不这种情况，DOCTYPE是用于激活使用HTML语法编写的文档的标准模式。 浏览器已经支持了<!DOCTYPE html>。

旧有的标签生成器并不能生成我们期望的短DOCTYPE。 为了支持它们，<!DOCTYPE html SYSTEM "about:legacy-compat">也是被HTML语法所允许的。

HTML4.0，HTML4.01，XHMTL1.0，还有XHTML1.1的严格DOCTYPE也允许（但不提倡）在HTML语法中使用。

在XML语法中，任何一种DOCTYPE声明都是可以使用的，或者也可以被省略。带有XML媒体类型的文档一直都会被以标准模式处理。

2.3 MathML and SVG

HTML语法允许在文档中使用MathML和SVG元素。一个math或者svg开始标签会使HTML解析器切换到一个特殊的插入模式。 这种模式下会将元素和属性放置在合适的命名空间中，给那些有大小混杂的元素或者属性做大小写修整，以及支持XML中的空元素语法。 这种语法仍然是大小写不敏感的，并且属性允许使用与HTML元素相同的语法。命名空间声明可以被省略。CDATA片段在插入模式中也是被支持的。

有些MathML和SVG标签会使解析器切回到"HTML"模式，例如mtext和foreignObject，因此你可以使用HTML元素或者一个新的math或者svg元素。

举例来说，一个非常简单的文档，使用最少的语法功能，可以是这样：

<!doctype html>
<title>SVG in text/html</title>
<p>
 A green circle:
 <svg> <circle r="50" cx="50" cy="50" fill="green"/> </svg>
</p>

2.4 杂项

还有一些语法调整值得一提：

  • 命名字符引用&lang;（数学左尖括号）和&rang;（数学右尖括号）分别从U+2329和U+232A调整为U+27E8和U+27E9。

  • 现在新添了很多命名字符引用，包括MathML中的所有命名字符引用。

  • Void元素（HTML4的EMPTY元素）允许拥有结尾的斜线。

  • 与HTML4相比，在更多情况下与符号（&）可以被左反转义。

  • 属性必须以至少一个空白字符分隔。

  • 没有值的属性可以只写属性名，而省略等号和值。即使属性不是一个布尔类型属性也可以这样做。（大家都认为HTML4允许布尔类型的属性值可以被省略。而对于枚举类型的属性则相反，HTML4允许只使用属性值而省略属性名。但是这并没有被浏览器所支持。）

  • 相比HTML4，HTML5中省略引号的属性值可以使用更多的字符

  • HTML解析器不再做任何属性值中空白字符的标准化；举例来说，在属性id的值开头和结尾的空白字符不再会被忽略掉（因此这样做是非法的），并且换行字符可以在input元素的value属性中使用了，不再需要使用字符引用了。

  • optgroup元素的关闭标签现在是可选的

  • colgroup 元素的开始标签现在是可选的，HTML解析器可以推断出它们。

3 语言

这一章分为几个小节来更加清晰地展示HTML5与HTML4的各种不同。

3.1 新元素

下列元素被引入，来优化HTML结构

  • section 表示一个普通文档或者一个应用章节。它可以用于 聚合 h1, h2, h3, h4, h5, and h6 元素，标明文档的结构。

  • article表示一个文档中一段独立的内容， 比如一篇博客或者报纸上的一篇文章。

  • main可以用作其他元素容纳显著内容的容器，例如页面的主要内容。 在W3C HTML5和W3C HTML5.1中，只文档中只允许有一个main元素。

  • aside代表一段与页面中其他内容仅仅有轻微关联的内容。

  • 在WHATW HTML规范中，hgroup代表章节的头部。

  • header代表一组引导性或者导航性的帮助信息。

  • footer代表一个章节的页脚，可以包含关于作者、版本等信息。

  • nav 代表一篇文档的导航章节。

  • figure代表一段独立的流内容，特别指一些从主文档流中分离出来的独立单元。

    <figure>
 <video src="example.webm" controls></video>
 <figcaption>Example</figcaption>
</figure>

    figcaption 可以用作标题（可选）

然后还有一些其他的新元素

  • video 和 audio 代表多媒体内容。这两个元素都提供了相应的API，因此网页应用开者可以开发他们自己的用户界面，同时也可以使用浏览器所提供的用户界面。当多种类型的流媒体可提供使用时， source元素可以与这些元素一同使用。

  • track 为video提供了文本字幕

  • embed 用作插件内容

  • mark代表一串被标记或者被高亮的文本，因为它可能与其他文本有所关联，所以用来表达引用或参考。

  • progress代表一项任务的完成情况，例如下载文件或者执行一系列耗时操作。

  • meter 代表度量，例如磁盘使用情况。

  • time代表日期和（或）时间

  • WHATWG HTML和W3C HTML5.1有data标签。它允许使用机器可读值来注解内容。

  • dialog用于显示一个对话框。

  • ruby, rt, 和 rp 允许ruby注解标记

  • bdi 代表一个文本应该和它周围的文本隔离开来，用来完成双向文本格式。

  • wbr 代表一个折行的时机。

  • canvas用来即时渲染动态位图图像，例如图表或游戏。

  • menuitem`代表用户可以触发的弹出菜单中的一个命令。

  • details代表附加信息或者控件，用户可以按需获取。 summary元素提供了它们的汇总、说明或说明。

  • datalist 与input的新属性list 一同使用可以用来构造下拉列表：

    <input list="browsers">
<datalist id="browsers">
 <option value="Safari">
 <option value="Internet Explorer">
 <option value="Opera">
 <option value="Firefox">
</datalist>

  • keygen代表密钥对生成器控件。

  • output代表一些种类的输出，例如来自使用脚本计算出的结果。

input 元素的type 属性现在可以使用以下新值：

  • tel
  • search
  • url
  • email
  • datetime
  • date
  • month
  • week
  • time
  • datetime-local
  • number
  • range
  • color

引入这些新类型的值是因为客户端可以提供用户界面，例如日历选择器或者是与用户的地址薄集成在一起，或者是提交一个固定格式的数据到服务器上。这能带给用户更好的体验，因为在数据被提交到服务之前就被校验而使用更少等待反馈时间。

3.2 新属性

我们引入了给HTMl4中已有的一些元素添加了新的属性。

  • WHATWG HTML和W3C HTML5.1中 a 和 area元素有了一个新属性download。WHATWG HTML标准中还引入了属性ping。

  • 为了与元素a和元素link保持一致，元素area现在也拥有属性hreflang, type and rel

  • 元素base现在也可以拥有一个target属性，主要是为了和a元素保持一致。(这个已经被广泛的支持了。)

  • meta元素现在拥有charset属性。这早已经是被广泛支持的属性，以一种很漂亮的方式给文档指定character encoding。

  • 在WHATWG HTML和W3C HTML5.1规范中，table元素拥有一个属性sortable， th元素拥有一个属性 sorted。这两个新属性为table提供了新的方式去排序table的列。

  • 新属性autofocus可以使用在 input (除了type属性的值是 hidden), select, textarea和 button元素. 它提供了一明确的方式，当页面加载过程中来聚集在表格控件上。相对于使用脚本来聚焦，使用这个特性可以增强用户体验。举例来说，如果用户不喜欢还可以把它关掉。

  • 新属性placeholder 可以使用在input和textarea元素上。它代表一个提示信息，用来帮助用户填写数据。

    <input type=email placeholder="a@b.com">

  • 新属性form可以使用在input, output, select, textarea, button, label, object and fieldset元素上，允许控件们与一个表单关联在一起。这些元素现在可以被放置在页面的任何一个位置，而不再只能是 form元素的子元素。

    <table>
 <tr>
  <th>Key
  <th>Value
  <th>Action
 <tr>
  <td><form id=1><input name=1-key></form>
  <td><input form=1 name=1-value>
  <td><button form=1 name=1-action value=save>✓</button>
      <button form=1 name=1-action value=delete>✗</button>
 ...
</table>

  • 新属性required可以应用在 input (除了type属性值是 hidden, image或者是某些按钮例如 submit), select and textarea。这个属性表示用户必须填写这个控件才能提交表单。对于select, 第一个option元素必须使用空值，当作提示信息。 

    <label>Color: <select name=color required>
 <option value="">Choose one
 <option>Red
 <option>Green
 <option>Blue
</select></label>

  • 元素fieldset现在可以设置属性 disabled。当被设定了disabled，这会使它的所有子元素都被置为失效(不包含legend元素的子元素)，并且name属性可以被脚本访问了。

  • 元素input拥有新的属性来限制用户输入： autocomplete, min, max, multiple, pattern and step. 前文提到过 list属性可以和 datalist元素一起使用。 现在还可以为type=image类型的input元素设定width and height属性来指定图片的尺寸。

  • 元素input和textarea元素现在有一个新属性，叫dirname。这个属性是用来设定控件输入值的方向性，当用户提交表单时也会此项数据也会被同时提交。

  • 元素textarea也拥有了两个新属性： maxlength和wrap分别控制着输入值的最大长度和提交数据的换行为。

  • 元素form现在拥有新属性novalidate，可以用来去掉表单提交时的校验功能。(例如，表单不管怎样都可以被提交)

  • 元素input和button现在拥有新属性 formaction, formenctype, formmethod, formnovalidate，和 formtarget。如果这些属性存在，它们会覆盖form的action, enctype, method, novalidate，和target属性

  • 在WHATWG HTML and W3C HTML5.1标准中， 元素input和textarea拥有一个inputmode属性。

  • 元素menu拥有两个新的属性： type和label。它们允许元素变成一个经典的用户界面中的菜单。同时，与全局属性contextmenu相结合，可以提供上下文菜单。

  • 在WHATWG HTML and W3C HTML5.1, 元素button拥有menu属性，可以和弹出菜单同时使用。

  • 元素style拥有一个新属性scoped，可以使用激活有作用域的样式表。在这样的style元素中的样式规则仅仅应用在本地的DOM树。

  • 元素script拥有一个新属性 async，可以影响到脚本的加载和执行。

  • 元素html现在有了一个新属性叫 manifest，与离线Web应用API相结合使用来指定应用的缓存。

  • 元素link拥有一个新的属性叫 sizes。它可以与 icon一同使用(通过设置rel属性；可以用于设定favicons），来标识引用的icon的尺寸，因此可以使用不同尺寸的icon。

  • 元素ol有一个新的属性，叫作 reversed。当它存在时，标识着列表的顺序是降序。

  • 元素iframe现在拥有三个新的属性，叫作 sandbox, seamless，和srcdoc ，允许沙盒内容，例如博客评论。

  • 元素object拥有一个新的属性叫typemustmatch，允许更加安全地嵌入外部资源。

  • 元素img有一个新属性叫 crossorigin，用于跨域加载图片并成功加载后，允许图片的数据可以被canvas API所使用。在 WHATWG HTML和W3C HTML5.1中，元素script也拥有一个新属性 crossorigin，允许script 执行出错将会汇报到onerror来通知相关的出错信息。在WHATWG HTML和W3C HTML5.1的link元素也有crossorigin属性。

  • 在WHATWG HTML， 元素img有一个新的属性叫srcset支持多个图片，不同的图片应用于不同大小的视图。

一些HTML4中的属性现在可以应用到所有元素了。这些属性被称为全局属性：accesskey, class, dir, id, lang, style, tabindex和title. 额外的，XHTML 1.0只允许xml:space在某些元素上使用，现在也可以在所有元素上使用了。

现在还有一些新的全局属性：

  • 属性contextmenu可以用来指向Web开发人员提供的上下文菜单。

  • 属性draggable和dropzone 可以与新drag & drop API。

  • 属性hidden标识一个元素还不是，或者再也不相关的。

  • WHATWG HTML and W3C HTML5.1拥有inert属性，用来使得dialog元素进入模态。（译者注：所谓模态对话框，就是指除非采取有效的关闭手段，用户的鼠标焦点或者输入光标将一直停留在其上的对话框。非模态对话框则不会强制此种特性，用户可以在当前对话框以及其他窗口间进行切换。）

  • role和aria-*属性集合可以用于辅助科技。这些属性在WHATWG HTML和W3C HTML5/W3C HTML5.1中略有不同。

  • 属性spellcheck允许用于提示内容是否可以是否拼写校验。

  • 属性translate用于提示翻译器该元素的内容是否应该被翻译

HTML把HTML4中使用形式为onevent的事件处理属性处理成全局属性，并且添加了一些新的事件处理属性给新添加的事件；例如，play事件的onplay事件处理属性，可以供多媒体(video and audio)元素的API使用。

3.3 改变了的属性

在HTML中，以下这些元素被略微修改了它们的含义，以更好的反映出它们应当在Web中使用，或者使它们更易于使用。

  • 元素address现在的作用域是它最近的article或body祖先元素

  • 元素b现在代表一段文本，仅仅是用于功利性目的来引起用户的注意，并不是为了传达任何重要性，而且也没有更换语音或情绪的含义。 例如文档摘要中的关键词，审查中产品名字，本文驱动的可交互软件中的可操作词，或者一篇文档的引言。

  • 元素 cite 现在只代表一个作品的标题（例如，一本书，一篇论文，一篇散文，一首诗，一份成绩单，一首歌，一份脚本，一个电影，一个电视节目，一个游戏，一尊雕塑，一幅画，一场戏剧，一场演出，一场歌剧，一场音乐会，一场展览，一份法律案例报告等等）。特别的，在HTML4的例子中它被用作标记一个人的名字，这一点两者已经不再是一致的了。

  • 元素 dl 现在表示一个name-value对的联合列表，不再表示适合表示一个对话。

  • 元素 hr 现在表示一个段落级别的主题间断。

  • 元素 i 现在表示一段更换声音或情绪的文字，或者是将在某种意义上有所不同的文字从普通文字中标识出来，例如生物学上的分类，一个科技术语，另一种语言中的熟语，一个想法，或者一个使用西文的船名。

  • 对于 label 元素，浏览器应该不再把焦点从label上移动到控件上，除非这样的行为是底层平台用户界面的标准

  • 元素 menu 被重新定义了，对工具栏和弹出菜单更好用。

  • 元素 noscript 不再表示如果客户端不能支持文档中 script 元素所使用的脚本才渲染的此元素。

  • 元素 s 现在表示内容不再准确，或者不再相关。

  • 元素 script 现在可以用于脚本或者自定义数据块。

  • 元素 small 现在表示次要评论，例如附属细则。

  • 元素 strong 现在表示重要，而不是加粗强调。

  • 元素 u 现在表示一段文字，它带有说不清意义却又明确地被渲染的非文本性注解。例如，把一段文字标记为一个中文特有的姓名（一个中文特有的姓名标签），或者把一段文本标记为拼写错了的。

  • 元素td 和 th 的属性 colspan 必须大于0。

  • 当元素 area 处于圆状态时，它的属性 coords 不再允许使用半径的百分比了。

  • 元素 object 的属性 data 不再和codebase属性相对的了。

  • 元素li 的属性 value 不再是弃用的，因为它会被不显示出来。 元素 ol 的属性 start 和 type 也是一样。

  • 全局属性 style 现在总是使用CSS样式语言。

4 内容模型

内容模型定义了元素可以怎样进行嵌套，即一个指定的元素可以拥有哪些子（子孙）元素。

从整体上看，HTML4有主要两类元素“inline”（例如，span， img，text），和 "block-level" (例如 div, hr, table). 有一些元素则两种不属于这两类中的任何一个。

一些元素允许包含“inline”元素 （例如， p), 有些可以包含 "block-level" 元素 （例如 body)， 有些则两种都可以包含（例如 div) 而其他的元素两类元素都不可以包含，却可以包含一些特殊的元素。（例如，dl, table）， 或者完全不允许包含子元素（例如， link, img, hr）

注意区别元素的类别和它的内容模型的类别。例如，p 元素本身是一个"block-level"元素， 但是它的内容模型却是"inline"。

更加令人迷惑的是HTML4在Strict模式、传统模式和Frameset模型下拥有不同的内容模型规则。举例来说，在Strict模式下，body 元素可以只能包含"block-level"元素，但是传统模式下 "inline" 和 "block-level"两类元素都是可使用的。

再一次令人迷惑的是CSS使用元素的"block-level"和"inline-level"属性来构建元素的视觉模型，这和CSS的'display'属性相关并且和HTML的内容模型规则一点关系都没有。

HTML没有把"block-level" 或者 "inline"纳入内容模型规则，来减少与CSS间的混乱。但是，HTML5却有比HTML4更多的类别categories，元素可以其中的几个、一个，或者一个都不是。

  • 原数据内容，例如：link, script。

  • 流内容，例如：span, div 和文本。这个大概和HTML中的"block-level" 加上 "inline"。

  • 章节内容，例如：aside, section.

  • 标题内容，例如：h1.

  • 语句内容，例如：span, img和文本。这个大概就是HTML4中的"inline"。元素如果是语句内容，那么它也是流内容。

  • 嵌入内容， 例如：img, iframe, svg。

  • 交互内容，例如：a, button, label。交互内容是不允许再包含其他元素的。

由于与HTML4有广泛的不同，HTML5再有那些只能包含HTML4中的"block-level"元素的元素了；例如，body 允许包含流内容。因此，相对于HTML4 Strict模式，这与HTML4的传统模式更接近。

进一步的改变包括：

  • 元素 address 现在可以包含流内容，但是不能包含标题内容子孙元素、章节内容子孙元素以及header, footer, 或者是 address 子孙元素.

  • HTML4 允许 object 出现在 head中。 HTML则不允许。

  • WHATWG HTML 允许 link 和 meta 成为 body 的子孙元素，如果它们使用了microdata属性。

  • 元素 noscript 在HTML4中是一个 "block-level" 元素，但是在HTML是一个语句内容元素。

  • 元素 table, thead, tbody, tfoot, tr, ol, ul 和 dl 在HTML可以是空结点。

  • Table 元素必须确保遵守了 table model (例如，两个单元格是不能重叠的)

  • 元素 table 元素不允许使用 col 元素作为子元素。但是，HTML解析器实现了元素colgroup ，因此这个改动并不会影响text/html 内容。

  • 元素 table 现在允许使用 tfoot 元素作为它的最后一个子结点。

  • 元素 caption 现在允许使用流内容作为子孙元素，但不能包含元素 table

  • 元素 th 现在允许流内容作为子孙元素，但不能包含 header, footer, 章节内容，或者是标题内容。

  • 元素 a 现在有 transparent 内容模式 ( 除非它不允许子孙元素是交互内容模型), 表示它使用与其父元素相同的内容模型。也就是说 a 元素现在可以包含例如 div 元素，如果它的父结点允许使用流内容。

  • 元素 ins 和 del 也拥有透明内容模型。HTML4也拥有这样类似的规则，但不能在DTD中表达。

  • 元素 object 也拥有透明内容模型，但它的流内容子结点必须在它的param子元素之后。

  • 元素 map 也拥有透明内容模型。如果元素area 是的祖先元素中有map 元素，那么认为它是语句内容的。这意味着它可以不是map的直接子元素。

  • 元素 fieldset 不再需要 legend 元素子元素。

5 APIs

HTML已经引入了许多新的API，并且扩展或弃用了某些API。

5.1 新API

HTML引入了不少新的API，来帮助实现Web应用。这些新API可以和新引入的元素一同使用。

  • 媒体元素(video 和 audio) 拥有控制重放、同步多媒体元素，时控文本轨道（例如字幕）等API。

  • form元素现在拥有一个新API来约束用户输入(例如， setCustomValidity() 方法)

  • 元素 commands 拥有一个新的API，供用户调用。

  • 离线Web应用拥有一个API来控制application cache.

  • 新的API registerProtocolHandler() 和 registerContentHandler() 允许Web应用将自己注册为使用某个指定的协议或媒体类型。

  • 编辑API可以和全局属性contenteditable结合使用。

  • 拖拽 & 放置 API 可以和属性 draggable 结合使用。

  • 新API接口 Location 可以暴露出文档URL的各个组件，并允许脚本控制文档的导航、重定向和重新加载。

  • 新API接口 History 暴露出会话历史，并且允许脚本不通过导航来更新文档的URL，因此应用可不必滥用分段片组件来实现Ajax风格导航。

  • 新的API来支持base64转义 (atob() 和 btoa() 方法).

  • 新的API来安排基于时间的回调 (setTimeout() 和 setInterval()).

  • 新的API来提示用户 (alert(), confirm(), prompt(), showModalDialog()).

  • 新的API来支持打印文档 (print()).

  • 新的API来处理搜索服务提供商 (AddSearchProvider() 和 IsSearchProviderInstalled()).

  • 对象 Window 已经被定义了。

WHATWG HTML 还有更多的新API，但是并不在W3C HTML5规范中，而是W3C中其他的单独规范。

  • 新的API来支持微数据。

  • 新的API来支持立即模式绘图 (元素的canvas 的 2d 上下文).

  • 新的API来支持跨域消息和信道通信 (postMessage() 和 MessageChannel).

  • 新的API来支持在后台运行脚本 (Worker 和 SharedWorker).

  • 新的API来支持客户端存储 (localStorage 和 sessionStorage).

  • 新的API来支持双向客户端-服务器端通信 (WebSocket).

  • 新的API来支持服务器端到客户端数据推送 (EventSource).

5.2 改变了的API

以下这些DOM Level2 HTML的功能以不同的方式发生了改变：

  • document.title 在获取时会去掉空白字符。

  • document.domain 现在可以设置了，这可以改变文档的有效脚本域。

  • document.open() 现在要么 带有少于两个参数而清空文档，要么带有三或四个参数则相当于 window.open() 如果在XML中按前者的方式调用，会抛出一个异常。

  • document.close(), document.write() 和 document.writeln() 在XML中会抛出异常。后两个函数现在支持可变参数；在文档的解析阶段，它们可以添加文本到文档的输入流中，意味着一次 document.open()的调用, 或者在某些情况下全部被忽略。

  • document.getElementsByName() 现在会返回所有带有name属性并且与参数匹配的HTML元素。

  • HTMLFormElement 的 elements 属性现在会返回一个 HTMLFormControlsCollection， 包括 button, fieldset, input, keygen, object, output, select and textarea 等元素. length 属性会返回elements所包含的结点数目。

  • HTMLSelectElement 的 add() 方法现在接受一个整数作为它的第二个参数。

  • HTMLSelectElement 的 remove() 方法，如果它的参数越界，现在会移除集合中的第一个元素

  • a 和 area 元素现在会将它们的href 属性字符串化。

  • 方法 click(), focus() and blur() 现在对于所有的HTML元素都可以使用了。

5.3 Document的扩展

DOM Level 2 HTML 有一个 HTMLDocument 接口，继承自 Document 并且提供了HTML特定成员。HTML把这些成员移动到了 Document 接口, 并且很多方面扩展了它。由于很多文档者使用 Document 接口，所以HTML的特定成员对于所有的文档现在还是可以用的，例如SVG文档。它们也有了一些新的成员：

  • location, lastModified 和 readyState 用来帮助管理资源元数据。

  • dir, head, embeds, plugins, scripts, commands, 以及一个通用名称的getter方法，可以用来访问DOM树中不同的部分。

  • WHATWG HTML使用getItems() 来获取微数据。

  • WHATWG HTML 和 W3C HTML5.1 的 cssElementMap 方法来完成CSS的 element() 功能。

  • WHATWG HTML 和 W3C HTML5.1 的 currentScript 方法返回当前正在执行的 script 元素或者是null。

  • activeElement 和 hasFocus 相应地可以用来决定哪个元素是当前的焦点和 Document 是否拥有焦点。

  • 新的编辑API designMode, execCommand(), queryCommandEnabled(), queryCommandIndeterm(), queryCommandState(), queryCommandSupported(), queryCommandValue()。

  • 所有的是事件处理都可以使用IDL属性来添加. 同样的， onreadystatechange 是一个特殊的事件处理IDL属性，只在Document上是有效的。

现有的修改了HTMLDocument的prototype的脚本应该是还可以工作的， 因为window.HTMLDocument现在返回了Document的接口对象。

5.4 HTMLElement的扩展

在HTML中，接口 HTMLElement 也得到了一些扩展：

  • translate, hidden, tabIndex, accessKey, draggable, dropzone, contentEditable, contextMenu, spellcheck 和 style 反映内容属性。

  • dataset 是一个便捷的功能，用来处理 data-* 属性。这些属性是以驼峰形式暴露出来的，例如， elm.dataset.fooBar = 'test' 设定了 elm 的 data-foo-bar 内容属性。

  • WHATWG HTML 为微数据提供了 itemScope, itemType, itemId, itemRef, itemProp, properties 和 itemValue。

  • click(), focus() and blur() 允许脚本模拟点击和移动焦点。

  • accessKeyLabel 提供了客户端给元素设定的快捷键，Web开发者可以通过accesskey来影响它。

  • 如果元素是可以编辑的，isContentEditable 会返回true。

  • forceSpellCheck() 会触发客户端来检查一个元素的拼写是否正确。

  • commandType, commandLabel, commandIcon, commandHidden, commandDisabled 和 commandChecked 是命令API的一部分。

  • 所有的事件处理都可以通过IDL属性来添加。

某些之前定义的HTMLElement的成员，在规范[DOM]中已经被移动到了的Element接口中

  • id 反映的是 id 内容属性。

  • className 反映的是 class 内容属性。

  • classList 是一个 className 属性的便捷访问器，这个对象返回暴露出来的方法(contains(), add()，remove()， 和 toggle())，以提供操纵元素的样式。

  • getElementsByClassName() 访求返回带有指定样式的元素列表。

5.5 其他接口的扩展

DOM Level 2 HTML中的某些接口获取了扩展

  • HTMLOptionsCollection 现在遗留调用者，setter方法创建器，和这些成员： add(), remove() 和 selectedIndex

  • HTMLLinkElement 和 HTMLStyleElement 现在实现了来自[CSSOM] LinkStyle 的接口。

  • HTMLOptionElement 现在有一个构造函数 Option.

  • HTMLInputElement 现在拥有成员 files, height, indeterminate, list, valueAsDate, valueAsNumber, width, stepUp(), stepDown()，以及表单约束性验证API成员， labels, 以及为文本域选择API使用的成员: selectionStart, selectionEnd, selectionDirection, setSelectionRange() 和 setRangeText()。

  • HTMLTextAreaElement 现在拥有成员 textLength，表单约束性验证API成员， labels以及文本域选择API成员。

  • HTMLButtonElement 现在拥有表单约束性验证API成员和 labels.

  • HTMLLabelElement 现在拥有成员 control.

  • HTMLFieldSetElement 现在拥有成员 type, elements 以及表单约束性验证API成员。

  • HTMLAnchorElement 现在拥有成员 relList, text, 并且实现了 URLUtils 接口。这个接口包括以下这些成员： href, origin, protocol, username, password, host, hostname, port, pathname, search, query 和 hash。 HTMLLinkElement 和 HTMLAreaElement 也拥有 relList IDL 属性。 HTMLAreaElement 也实现了 URLUtils 接口。

  • HTMLIFrameElement 现在拥有成员 contentWindow.

5.6 弃用的API

一些API现在被移除了，或者被标记为弃用

用于反映被弃用了的内容属性的IDL属性，也全部被弃用了。举例来说，HTMLBodyElement 的 bgColor IDL 属性反映了已被弃用的 bgcolor内容属性，那么它也被弃用了。

以下这些接口被标记为弃用了，因为相关的元素被弃用了： HTMLAppletElement, HTMLFrameSetElement, HTMLFrameElement, HTMLDirectoryElement and HTMLFontElement.

接口 HTMLIsIndexElement 现在被一共移除了，因为HTML解析器扩展了 isindex 标签到其他的元素。接口 HTMLBaseFontElement 也被移除了，因此元素没有实际效果。

以下这些HTMLDocument接口（已被移动到Document）的成员现在也被弃用了: anchors 和 applets.

  • The pushState and replaceState features have been changed based on implementation feedback in Firefox, and history.state has been introduced.

  • The action and formaction attributes are no longer allowed to have the empty string as value.

  • 新增了全局 dropzone 属性

  • A new bdi element was added to aid with user-generated content that may have bidi implications.

In addition lots of minor changes, clarifications, and fixes have been made to the document.

6.9 从2009-08-25到2010-03-04发生的变更

  • The dialog element has been removed. A section with advice on how to mark up conversations has effectively replaced it.

  • document.head has been introduced to provide convenient access to the head element from script.

  • The link type feed has been removed. alternate with specific media types is to be used instead.

  • createHTMLDocument() has been introduced as API to allow easy creation of HTML documents.

  • The details element now uses a new element summary for exactly the same reason.

  • The autobuffer attribute on media elements was renamed to preload.

A whole lot of other smaller issues have also been resolved. The above list summarizes what is thought to be of primary interest to Web developers.

  • HTML Canvas 2D Context

  • HTML5 Web Messaging
